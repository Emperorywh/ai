# AGENTS — 本仓库编码约束

> 本文件是 agent 通用执行约束的**唯一权威来源**（见 `Readme.md` §6.1 / §13）。任务或工作流特有的增量约束（如修改范围、状态机规则）写在对应任务文件中，此处不重复维护，避免日后改一处漏一处。

## 1. 沟通与语言

- 使用**简体中文**回复与撰写文档。
- 复杂或非显而易见的逻辑必须添加**简体中文多行注释**；自解释的简单代码不强求注释，避免噪声。
- 生成 commit message 必须使用简体中文。

## 2. 架构与边界

- **先理解架构，再编码**：先明确数据流、状态流与模块边界，再动手实现。
- **不跨层调用**：分层依赖方向为 `cli → application → core ← infrastructure`；`core` 不反向依赖任何层；`application` 经 `src/application/ports.ts` 窄接口依赖 `infrastructure`，不得直接 import 其实现类。
- **发现架构不合理时优先重构**，不要继续堆逻辑；若修复需要越过 `forbidden_paths`，应在 `.result.md` 的 `global_update_requests.issues` 中提出问题并把 `next_action` 设为 `needs-human`，由 Orchestrator 扩权后再继续，不自行越界。

## 3. 代码质量

- **不引入临时 patch** 解决结构性问题；网络重试、错误提示、环境能力检测等运行时容错必须作为显式错误处理或能力声明写入文档，不作为隐藏兼容逻辑存在。
- **不写巨型函数或巨型组件**：职责单一，必要时拆分。
- **不制造隐式状态**：状态归属与流转必须显式、可追踪。
- **不复制粘贴重复逻辑**：优先抽取复用，避免重复实现。

## 4. 工作纪律

- **不主动格式化无关代码**，只改当前任务范围内必须改的部分。
- **不自动启动浏览器测试**，除非用户明确要求。
- 每个任务只负责一个清晰目标，只完成当前任务，不提前实现后续任务。
- 每次修改后检查是否增加耦合、引入技术债或破坏架构一致性。

## 5. 验证

- 完成后运行必要验证：`npm run typecheck`、`npm test`、`npm run lint`（命令适用 layer 见 `docs/TESTING.md`）。
- 不自动跑浏览器测试；CLI 端到端在临时项目目录中验证。

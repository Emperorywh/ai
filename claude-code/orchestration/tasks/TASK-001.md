title: 建立唯一的规范编码与哈希边界
---

## 任务描述

### 可验证结果

系统具备唯一、可重算、跨平台稳定的规范编码与哈希能力。相同领域对象、相同 SPEC/TASK 语义和相同附件字节在允许的平台上得到相同摘要；未知字段、非法文本、非法路径或非规范输入会在进入证据链前被拒绝。

### 输入

- `SupervisedExecutionLoopSpec.md` 第 5.2、7、8.5、14.4、17 和 20.1 节。
- 当前项目文档、任务契约、候选指纹分别自行计算 SHA-256 的现状。
- 当前项目加载器直接按文件文本计算项目哈希，尚无统一 strict Schema、JCS 和路径规范协议。

### 输出

- 版本化 strict Schema 校验后的结构化对象规范编码与 SHA-256 摘要能力。
- SPEC/TASK 源文本的 UTF-8、无 BOM、LF 规范化及 source hash 能力。
- 完整正文参与的 SPEC/TASK contract projection 与 contract hash 能力。
- 二进制和文本附件按保存后原始字节计算摘要、长度和媒体类型的契约。
- Git 相对 POSIX 路径的 UTF-8、Unicode NFC、大小写折叠碰撞和目标平台可表示性校验能力。
- 覆盖规范编码、文本归一化、附件字节和路径拒绝规则的自动化测试，以及说明唯一哈希入口和版本升级规则的必要架构文档。

### 实现约束

- 所有结构化哈希必须先通过版本化 strict Schema，再按 JSON Canonicalization Scheme 编码；摘要固定为小写十六进制 SHA-256。
- 未知字段、重复规范键、非有限数字、非法 Unicode、Schema 外联合分支、BOM、NUL 和路径碰撞必须 fail closed，禁止清洗后继续。
- SPEC/TASK 只允许 CRLF/CR 到 LF 的换行归一化，其他正文字符不得改写；正文变化必须改变 contract hash。
- 附件不得做隐式换行或文本归一化。
- 数组保持领域规定顺序；不能为追求稳定而擅自排序具有业务顺序的数组。
- 领域与应用层只依赖规范 DTO 和哈希端口，不得依赖文件系统、Git 或 SDK 类型。
- 不保留旧哈希算法 fallback，不形成多个可互换的规范编码入口。

### 验证方式

1. 运行 `pnpm typecheck`、`pnpm lint`、`pnpm test`、`pnpm build`。
2. 正常路径：验证等价 LF/CRLF 的 SPEC/TASK 源文本得到相同 source/contract hash，且同一 strict 领域对象重复编码得到逐字节相同结果。
3. 正常路径：验证正文、结构化字段、附件原始字节或有序数组任一有效变化都会改变对应摘要。
4. 异常路径：验证 BOM、NUL、未知字段、非有限数字、非法 Unicode、非 NFC 路径、规范化碰撞、大小写折叠碰撞和平台不可表示路径均被明确拒绝。
5. 异常路径：验证调用方不能绕过 Schema 直接对任意对象签发规范摘要。

### 完成标准

- 上述规范编码、哈希和路径验证结果均已实现并可自动复现。
- 全部自动化验证通过，当前线性任务加载、Git 候选和状态持久化既有测试未被破坏。
- 仓库中不存在第二套规范哈希算法、宽松解析、旧算法 fallback 或通过删字段获得摘要的临时逻辑。
- 领域、应用和基础设施依赖方向清晰，可安全创建本 TASK 的独立 Git checkpoint。
- 代码库可在该 checkpoint 上继续实现后续契约编译能力。

### 回退边界

回退本 TASK 的 Git checkpoint 只移除统一规范编码、哈希和路径校验能力及其测试/文档，不改变当前 v6 队列、Agent 执行、Git 锁、候选隔离或提交行为。

请完整阅读 `orchestration/SPEC.md`，并结合当前代码库的架构、模块边界、数据流和状态流，将 SPEC 拆分为适合 Coding Agent 线性执行的 TASK。

本次只生成 TASK 文档，不实现业务功能，不修改业务代码。

## 拆分原则

1. 必须按“可验证结果”拆分，不得按时间、开发阶段、前端、后端、数据库或测试等技术工种机械拆分。
2. 每个 TASK 只交付一个高内聚结果，并具有明确的输入、输出、验证方式、完成标准和回退边界。
3. 每个 TASK 完成后，代码库必须处于一致、可运行、可验证的状态，并可以生成一个独立 Git checkpoint。
4. 测试、必要重构和相关文档必须包含在对应 TASK 中，不要单独创建“编写测试”“代码清理”“架构重构”等任务。
5. 如果现有架构无法正确支持目标结果，应将必要重构纳入最早需要它的 TASK，禁止临时 patch、重复逻辑、跨层耦合和兼容旧实现。
6. 先定义可验证结果，再用大约 20～60 分钟作为粒度检查标准，禁止按时间本身拆任务。
7. 如果一个 TASK 明显过大，应继续按照独立用户行为、状态转换、数据闭环、接口契约或异常场景拆分；拆分后的每个结果仍必须可以独立验证。
8. 如果拆分后的 TASK 无法独立验证，说明边界不成立，应重新设计。

## 线性执行规则

- TASK 按 ID 数字顺序严格线性执行。
- 使用 `TASK-001`、`TASK-002`、`TASK-003` 格式编号。
- 只有前一个 TASK 完成后，才能执行下一个 TASK。
- 每个 TASK 必须建立在此前所有 TASK 已完成的代码状态上。
- 后续 TASK 不得成为当前 TASK 完成或验证的必要条件。
- 任一 TASK 失败或阻塞时，后续 TASK 应保持未执行状态。

## TASK 文档格式

每个 TASK 必须严格采用以下格式，YAML 前置元数据只能包含 `id` 和 `title`：

---
id: TASK-001
title: 简洁描述本任务交付的结果
---

## 任务描述

### 可验证结果

描述 TASK 完成后，系统新增了什么可以直接观察和验证的能力。

### 输入

列出相关 SPEC 要求、现有模块，以及此前 TASK 已经交付的能力。

### 输出

列出本 TASK 需要交付的业务行为、领域能力、接口、状态变化、测试和必要文档。

### 实现约束

说明必须遵守的模块边界、依赖方向、状态不变量和架构要求。不要预设不必要的具体文件路径或实现细节。

### 验证方式

给出 Coding Agent 可以实际执行的验证步骤，包括相关命令、正常路径、关键异常路径和预期结果。

不得自动启动浏览器。需要界面验收时，只能记录为人工验收事项。

### 完成标准

列出可以明确判断 TASK 已完成的条件，至少保证：

- 可验证结果已经实现；
- 相关自动化验证通过；
- 没有破坏此前 TASK 的行为；
- 没有遗留临时逻辑或重复实现；
- 代码库可以安全进入下一个 TASK；
- 当前结果可以生成独立 Git checkpoint。

### 回退边界

说明回退本 TASK 的 Git checkpoint 时，只会移除本 TASK 新增的结果，不会破坏此前已经完成的能力。

## 工作流程

1. 阅读完整 SPEC。
2. 检查相关现有代码。
3. 梳理领域边界、数据流和状态流。
4. 提取 SPEC 中的全部可验证结果。
5. 将结果组织成严格线性序列。
6. 检查每个 TASK 的独立验证能力和 Git checkpoint 边界。
7. 检查每项 SPEC 要求是否被 TASK 完整覆盖。
8. 在 `orchestration/tasks` 中生成 TASK Markdown 文件。
9. 不保留与本次拆分结果无关的旧 TASK 文档。
10. 最终列出 TASK 顺序，并用一句话说明每个 TASK 的可验证结果。

如果 SPEC 存在无法从文档和代码推导、且会实质影响任务边界的关键缺失信息，不要自行发明需求。请停止生成受影响的 TASK，并列出需要确认的问题。

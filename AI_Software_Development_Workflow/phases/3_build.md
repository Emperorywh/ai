# 阶段 3：执行（Build）

> 角色：资深软件工程师。
> 目标：完成**一个** Task。完成 = 代码写好 + 测试通过 + 自检全绿。**做完即停，等用户人工验收。**
> 触发：用户说「推进 / 继续 / next」/ 粘贴【推进提示词】（见 `templates/entry-prompts.md`，每次同一句）。

## 输入

- `docs/FEATURES.md`（取「当前活动功能」→ 其目录）
- 该功能 **`PROGRESS.md`**（看「当前指针」，定位本次 Task = 第一个非 ✅）
- 该功能 `ROADMAP.md`（本 Task 行 + 所属 Milestone 的开发边界 / 验收标准）
- 该功能 `SPEC.md`（对照需求；技术栈 / 构建命令 / 共享契约从这里取，不要假设）
- 相关现有代码

## 第一步：判定风险等级（决定怎么干）

先读 Task 的「风险等级」，或自行判定：

| 等级 | 触发条件 | 你的行为 |
|---|---|---|
| 🟢 低 | 新增文件、纯函数、有测试覆盖的改动、UI 调整、文档 | **自主完成**，做完自测通过即可，不打断用户 |
| 🟡 中 | 改非公共模块、新增依赖、改配置 | 自主做，但**完成后明确汇报**改了什么、为什么 |
| 🔴 高 | 改架构、改公共接口、删数据/文件、数据迁移、破坏性变更 | **先出实现方案，等用户确认**再动手 |

判定不清时，**按高一级处理**（保守）。

## 低/中风险 Task 的工作流（自主）

1. 用 TodoList 把 Task 拆成几步，逐个做、逐个勾。
2. 边写代码边写测试（测试内建，见下）。
3. 写完**自己跑测试 / lint / typecheck**，失败就修，循环到绿。
4. 更新 `PROGRESS.md`。
5. 汇报：改了什么、测试结果，**然后停下等验收**。

## 高风险 Task 的工作流（需确认）

1. 先输出：`# 实现方案` / `# 修改文件列表` / `# 新增文件列表` / `# 风险分析（可能破坏什么、如何回滚）`。
2. **等用户确认**再编码。
3. 编码 + 自测同上。

## 测试内建（硬性要求）

- 写功能的同时写测试。不存在「先写完再补测试」。
- 改老代码前，**先给它补测试**（characterization test），再改。
- Task 完成的定义里**必须包含测试通过**。

## 自测循环（agent 最强的能力，必须用）

写完代码后**亲自执行**：
```
跑测试 → 失败 → 读报错 → 改 → 再跑 → ... → 全绿
跑 lint / typecheck → 有错就修
```
- 不要只给「测试建议」——要**实际运行并让测试通过**。
- 测试/构建命令从 SPEC/ROADMAP 取；找不到就查 `package.json` / `Makefile` / README，或问用户。

## 铁律

1. **只实现当前 Task**，不开发未来功能（YAGNI）、不跨 Task、不提前做下一个。
2. 只改该 Task「Claude Code 开发边界」允许的目录；越界前必须问。
3. 不随意改「已验收」模块——要改必须在实现方案里说明理由 + 更新其测试。
4. 优先保证可运行；保持代码风格与现有一致。

## 完成自检（全绿才算 Task 完成）

- [ ] 当前 Task 功能全部实现（边界内）
- [ ] 测试已写且**全部通过**（你亲自跑过）
- [ ] lint / typecheck 无新增错误
- [ ] 没有引入未使用的依赖 / 死代码
- [ ] `PROGRESS.md` 已更新（该 Task 标 ✅ + commit 短 hash + 当前指针前移 + 踩坑记入「近期注意事项」）
- [ ] （🔴 高风险）方案已获用户确认

## 完成后（人工闸门 —— 对应用户第 5、6 点）

- **做完单个 Task 即停**：commit（含 PROGRESS 更新）后，报告「做了什么 / 自测与验收结果 / 下一个 Task 是什么」，**交用户人工验收**。
- **不要自动做下一个 Task。** 用户验收 OK 后，再粘贴【推进提示词】（同一句）推进下一个。
- 🔴 高风险 Task：即便自测绿，也请用户确认后视为完成。

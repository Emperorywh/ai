# 阶段 1：理解 / 深度访谈（Understand）

> 角色：资深产品经理 + 系统分析师 + 评审者。
> 目标：**先理解问题，再谈方案**——通过持续多轮访谈把需求问透，产出落盘的 `docs/features/<功能名>/SPEC.md`。
> 触发：用户提出新功能 / 说「访谈」/ 粘贴【访谈提示词】（见 `templates/entry-prompts.md`）。

## 核心姿态：这不是全自动，是「访谈」

- **不要直接设计系统 / 数据库 / 架构**，也不要一上来就给方案。先问。
- **不要假设**——模糊就追问。
- 你是主动方：**挖用户没想到的难点**，而不是只被动澄清用户说的。
- 每轮控制在**可消化**的题量（建议 ≤5 题），但**轮数不设上限**，直到覆盖完整。

## 五大追问维度（逐轮轮转，全部要覆盖）

1. **技术实现**：技术栈 / 架构 / 数据模型 / 接口 / 外部依赖 / 部署环境。
2. **UI / UX**：核心交互流程 / 关键界面 / 状态机 / 加载·空·错误态 / 可访问性。
3. **边界与异常**：空数据 / 极值 / 并发 / 失败重试 / 权限 / 超时 / 兼容性 / 数据一致性。
4. **顾虑与权衡取舍**：成本 / 性能 / 安全 / 工期 / 可维护性；每个有分歧的点给出 A vs B 的取舍与**推荐默认值**。
5. **隐性难点**：容易被低估的复杂度、最容易返工的地方、依赖链上的不确定项、合规/许可/数据来源。

## 每轮的节奏

```
问 → 听 → 小结（复述当前理解 + 标记已决/未决）→ 继续追问下一维度
```

- 每个需要拍板的议题，当场记入「决策日志」（议题 / 决策 / 理由），后续直接进 SPEC §0。
- 带默认值的待确认项，给出 **【默认】** 建议，用户无异议即按默认推进。

## 现有项目 / 接手既有代码（先做这步）

1. 读现有 README / CLAUDE.md / `docs/`。
2. 扫目录结构，定位关键模块与「共享契约 / 禁改边界」。
3. 把对现状的理解写进 `SPEC.md` 的「现状」章节，再进入需求澄清。

## 何时结束访谈（全部满足才停）

- 项目目标能**一句话**说清？
- MVP 边界明确（做什么 + 不做什么）？
- 五大维度都覆盖，无明显遗漏的阻塞项？
- 决策日志（D1..Dn）齐全，每个开放问题都有默认值？
- 没有遗留的「待确认」会阻塞开发？

## 输出

1. **写 `docs/features/<功能名>/SPEC.md`**（用 `templates/spec.template.md` 结构）：
   - 不要营销语言、不要模糊描述。
   - **禁止**「后续考虑 / 可以扩展 / 视情况而定」——出现就是没做完。
   - 必含：决策日志（§0）、现状、目标、非目标、用户角色/场景、功能需求、非功能需求、技术栈、架构/关键设计（粗粒度）、数据模型（粗粒度）、API（粗粒度）、**边界情况与陷阱清单**、**风险与权衡**、MVP 范围、**待确认（带默认值）**。
   - 数据模型 / API 只需**粗粒度**，细节在 build 阶段按 Task 落地，过早冻结会僵化。
2. **注册到 `docs/FEATURES.md`**：在功能清单新增一行；把「当前活动功能」指针指到本功能。

## 完成自检（全绿才算阶段 1 完成）

- [ ] `SPEC.md` 已写入，无「后续考虑」类模糊表述
- [ ] 决策日志齐全，五大维度都覆盖
- [ ] 边界情况清单、风险与权衡、待确认（带默认值）三节都有实质内容
- [ ] MVP 范围与非目标范围明确
- [ ] 没有会阻塞开发的「待确认」项
- [ ] （接手既有代码）现状章节已写
- [ ] `FEATURES.md` 已注册并指向本功能

## 完成后（人工闸门）

告诉用户："阶段 1 完成，SPEC 已落盘于 `docs/features/<功能名>/SPEC.md`，已设为当前活动功能。请确认 SPEC；确认后说『规划』进入阶段 2。"
**不要**直接进入阶段 2——让用户先确认范围。

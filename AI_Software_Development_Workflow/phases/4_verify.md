# 阶段 4：验证（Verify）

> 角色：Staff 工程师 + QA。
> 目标：验证当前 Task（或整个 Milestone）**确实达标**；发现问题就修，不只是报告。
> **分工**：agent 负责可自动化的验证（测试/静态检查/对照验收/P0 自修）；**主观项（视觉、外部行为）由用户人工验收**（在阶段 3 做完即停时进行）。

## 输入

- 代码 + 测试（阶段 3 产出）
- 该功能 `ROADMAP.md` 的验收标准 + `SPEC.md`
- 该功能 `PROGRESS.md`

## agent 验证内容（全部实际执行，不要只看代码）

1. **跑全量测试**（不只是当前 Task 的）：确认没有回归。
2. **lint / typecheck / build**：全部通过。
3. **Code Review**（可派 subagent 并行多维度）：架构、类型安全、性能、重复逻辑、潜在 Bug、边界情况、安全、可维护性。
4. **对照验收标准**逐条核对（尽量可编程断言）。

## ⚠️ 发现问题 → 回路

- **P0（必须修复：Bug、安全、回归、验收标准未达成）**
  → **回到阶段 3**，建子任务修复 → 自测 → 回阶段 4 再验证。不要把 P0 留给用户。
- **P1（建议修复）** → 能顺手修就修；修不了记入 `PROGRESS.md` 踩坑区 / 新增 Task。
- **P2（优化建议）** → 记入踩坑区或 Task 表，不在本阶段处理。

## 交人工验收的部分

agent 做不到可靠自动判定的，**做到可验证点即停、交用户 Review**：
- 视觉美学 / UI 效果（截图给用户看）
- 真实外部行为 / 真实性能（真实 GPU、真实数据）
- 合规 / 许可 / 翻译准确性 / 政治敏感内容

## 输出（写回 `PROGRESS.md` 的「近期注意事项」）

```
【Task NN 验证记录】
- 全量测试：（是否通过、回归）
- 静态检查：（lint / typecheck / build）
- Code Review：（P0 / P1 / P2 + 是否已修）
- 验收标准核对：（逐条 ✓ / ✗）
- 结论：PASS / 回阶段3（列 P0）
```

## 完成自检

- [ ] 全量测试通过（含回归）
- [ ] lint / typecheck / build 通过
- [ ] 所有 P0 已修复并复测
- [ ] 验收标准逐条达成（或主观项已交用户并记录）
- [ ] 验证记录已写入 `PROGRESS.md`
- [ ] `PROGRESS.md` 状态正确（PASS 时为 ✅；P0 回路时为 🔄）

## 完成后

- PASS → 该 Task 在 `PROGRESS.md` 已是 ✅（阶段 3 收尾标定，这里复核）；**报告用户人工验收要点**（哪些主观项要看），等用户确认后再推进下一个 Task。
- 有 P0 → 回阶段 3 修复（该 Task 在 PROGRESS 回退为 🔄），修完再来自检。

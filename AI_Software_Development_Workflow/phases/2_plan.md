# 阶段 2：规划（Plan）

> 角色：技术负责人。
> 目标：把 SPEC 拆成**可独立验证的** Milestone 和 Task，落盘 `ROADMAP.md` + `PROGRESS.md`。
> 触发：用户说「规划 / 拆分 / 出路线图」/ 粘贴【规划提示词】（见 `templates/entry-prompts.md`）。

## 输入

- `docs/FEATURES.md`（取「当前活动功能」→ 其目录 `docs/features/<功能名>/`）
- 该功能 `SPEC.md`

## 输出（写到该功能目录下）

- `docs/features/<功能名>/ROADMAP.md`：路线图 + 每 Milestone 详细设计 + Task 清单表（内联）+ 实施风险报告
- `docs/features/<功能名>/PROGRESS.md`：初始化进度表（Task 状态全 ⬜、「当前指针」指向 Task 01）—— **后续开发的驱动器**

## Milestone 拆分原则（可独立验证）

- **按用户价值拆（纵向切片），不按技术层拆。**
  - ❌ 前端阶段 / 后端阶段 / 数据库阶段
  - ✅ 「用户能注册登录」/「用户能下单」/「订单能发货」
- 每个 Milestone 必须：**可独立开发、可独立运行、可独立测试、可独立验收**。
- 高风险 / 不确定的**优先排前面**（早验证）。

每个 Milestone 写清（对齐参考项目 ROADMAP）：目标、用户价值、实现范围、不包含内容、涉及目录、依赖关系、**风险验证项**、验收标准、Definition of Done、**Claude Code 开发边界（✅允许 / ⚠️受限 / 🚫禁止目录）**。

## Task 拆分原则（可独立验证的原子单元）

- 度量单位是**「可独立验证的原子单元」**，不是工时。
- 每个 Task：可单独开发、可单独测试、可单独提交（一次 commit）。
- 粒度参考：一个 Task 应能在一轮 agent 工作里**做完并自测通过**（典型 ≈ 半天到一天）。
- **太大就拆。** 依赖关系构成 **DAG（无环）**，写在 Task 表里。
- 每个 Task 标 **风险等级（🟢低 / 🟡中 / 🔴高）**——直接决定阶段 3 的自主性。

## 完成自检

- [ ] `ROADMAP.md` 的 Milestone 都按用户价值拆分、可独立验证
- [ ] 每个 Milestone 有验收标准、DoD、**Claude Code 开发边界**
- [ ] Task 表粒度合适（能一轮做完 + 自测）、依赖 DAG 无环、每 Task 标了风险等级
- [ ] 实施风险报告（返工点 / PoC 时机 / 推迟项 / 适合自动化 / 需人工 Review）已写
- [ ] `PROGRESS.md` 已初始化（Task 状态表 + 当前指针指向 Task 01）
- [ ] `FEATURES.md` 状态更新为「规划中」

## 完成后（人工闸门）

输出「推荐开发顺序」，告诉用户："阶段 2 完成，ROADMAP + PROGRESS 已落盘。请确认规划；确认后粘贴【推进提示词】（或说『推进 / 继续』）从 Task 01 开始。"
**不要**自动开始推进——让用户先确认规划。

# AI 软件工程工作流（Claude Code 版）

> 把 agent 当成**有持久记忆、能自主执行+自测+修复循环**的工程师，
> 但**关键节点停下来等人**——不是全自动流水线。
> 一个项目里**每个功能（feature）各自走完整流程**，互不干扰。

---

## 0. 这个文件是干什么的

把本文件放到**项目根目录**，Claude Code 每次启动会自动加载它。它定义了：

- **按功能自治**：每个功能在自己的 `docs/features/<功能名>/` 下有一套完整 SPEC/ROADMAP/PROGRESS。
- **三段式可复用提示词 + 人工闸门**：① 访谈 → ② 规划 → ③ 推进（每次同一句），段与段之间、Task 与 Task 之间都**停下等人**。
- **进度驱动**：靠 `docs/FEATURES.md`（功能注册表 + 当前活动功能指针）+ 每个功能的 `PROGRESS.md` 串起全流程。
- 开发的**风险分级**（哪些 agent 自主做、哪些要人确认）。
- **测试内建**与**自测循环**的硬性要求。
- **版本控制**约定（分支 / commit / PR）。
- 何时回到上一阶段（**回路**）。

适用对象：Claude Code。充分用上文件落盘、CLAUDE.md 记忆、`写代码 → 跑测试 → 看结果 → 修复` 的自循环、TodoList。

## 1. 按功能自治：五个阶段 + 一条回路 + 人工闸门

```
        ┌───────────────────────────────────────────────┐
        │   发现 P0/验收FAIL/人工验收不通过 → 回「执行」修复 │
        ▼                                               │
[1 理解] → [2 规划] → [3 执行] → [4 验证] → [5 集成]
 访谈        规划       推进        验证        发布
  🚪停        🚪停       🚪停(每Task)              🚪停
```

**每个功能都走这条线，但段之间有闸门（🚪 = 停下等人）：**

| 阶段 | 触发提示词 | 产出 | 闸门 |
|---|---|---|---|
| 1 理解 | **访谈**（新功能说需求即可） | `docs/features/<功能名>/SPEC.md` + 注册到 `FEATURES.md` | 停，等确认 SPEC |
| 2 规划 | **规划**（「规划/拆分/出路线图」） | `ROADMAP.md` + `PROGRESS.md`（指针指向 Task 01） | 停，等确认规划 |
| 3 执行 | **推进**（「推进/继续/next」，**每次同一句**） | 代码+测试 + 更新 `PROGRESS.md` + commit | **停，等人工验收**；OK 后再粘同一句做下一个 Task |
| 4 验证 | （含在推进收尾 / 或单独「验证」） | 验证记录写回 `PROGRESS.md` | P0 回阶段 3；主观项交人 |
| 5 集成 | 「发布/集成」 | PR / 归档 + 更新 `PROGRESS.md`、`FEATURES.md` | 未授权不 push/不开 PR |

**回路规则（关键）：**

- `4 验证` 发现 P0 / Bug / 回归 → 回 `3 执行` 修复，**不重走全流程**。
- `5 集成` 验收 FAIL → 实现问题回 `3 执行`；需求理解错则回 `1 理解` 更新 `SPEC.md`。
- 任何阶段发现**需求本身有问题** → 回 `1 理解`，更新 SPEC。
- **人工验收不通过** → 回 `3 执行` 按用户反馈修。

每个阶段不是「说一段话」，而是「**产出一份落盘文件 + 通过自检 checklist + 在闸门停下**」。

### 三段式提示词（驱动全流程）

三段提示词都在 `templates/entry-prompts.md`（每段有「👇 复制这段」框，可直接粘贴）。也可用触发词：

- **① 访谈**：用户提出新功能 / 说「访谈」。→ 深度多轮访谈（见阶段 1），落盘 SPEC，停下。
- **② 规划**：用户说「规划 / 拆分 / 出路线图」。→ 读 SPEC 拆可独立验证的 Milestone/Task，落盘 ROADMAP+PROGRESS，停下。
- **③ 推进**：用户说「推进 / 继续 / next」或粘贴【推进提示词】（**每次同一句**）。→ 读 `FEATURES.md`→当前功能→`PROGRESS.md` 定位当前 Task → 实现+自测 → 更新 PROGRESS → commit → **停下等人工验收**。

> 「推进」每次同一句之所以成立：它**不内嵌任何功能特定值**，项目名/技术栈/构建命令/边界全部从该功能 SPEC/ROADMAP/PROGRESS 读，只靠 `FEATURES.md` 的「当前活动功能」指针定位。

## 2. 核心原则（必须遵守）

### 2.1 落盘优先（agent 的记忆 = 文件）
- 每个阶段的产物**必须写入 `docs/` 下的文件**，不能只留在对话里。
- 下游阶段**读文件**获取上游产物，不靠对话历史。
- 对话会被压缩/清空，文件不会。`docs/` 就是这个项目的知识库。

### 2.2 按功能自治
- 一个项目可同时有多个功能；每个功能在 `docs/features/<功能名>/` 下有独立的 SPEC/ROADMAP/PROGRESS。
- `docs/FEATURES.md` 是项目级注册表，记录所有功能与**「当前活动功能」指针**。
- 三段提示词都靠这个指针定位「现在做哪个功能」。切换功能 = 改指针（或说「切到 <功能名>」）。

### 2.3 人工闸门（不是全自动）
- 段与段之间、Task 与 Task 之间都**停下等人**：确认 SPEC → 确认规划 → 每个 Task 人工验收后再推进。
- 做完一个 Task：commit + 更新 PROGRESS + 报告，**然后停**，不自动做下一个。
- 需求不清楚、方案有分歧、高风险改动 → 先问，不动手。

### 2.4 单一职责
- 每个阶段只做一件事，有明确输入文件和输出文件。
- 执行阶段不重新设计架构（架构问题记到 SPEC 决策日志 / 回规划）；验证阶段不写新功能。

### 2.5 风险分级决定自主性
- 🟢 **低风险**（新文件、纯函数、有测试覆盖的改动、UI、文档）→ agent **自主完成并自测**，不打断用户。
- 🟡 **中风险**（改非公共模块、加依赖、改配置）→ 自主做，**完成后汇报**改了什么、为什么。
- 🔴 **高风险**（改架构、改公共接口、删数据/文件、迁移、破坏性变更）→ **先出方案、等确认**再动手。
- 判定不清时按**高一级**处理。详见 `phases/3_build.md`。

### 2.6 测试内建，不是后置
- 写功能的同时写测试。一个改动「完成」的定义包含**测试通过**。
- 不存在「先写完代码、以后再补测试」。

### 2.7 自测循环（agent 最强的能力，必须用）
- 写完代码 → **自己跑测试/lint/typecheck** → 失败就修 → 再跑 → 直到全绿。
- 不要写完就甩给用户一句「测试建议」，要**实际执行并让测试通过**。

### 2.8 不随意改已验收模块
- 改 `PROGRESS.md` 标记为已验收的模块时，必须在实现方案里说明**理由**并更新对应测试。
- 改 `PROGRESS.md` 是为了**如实记录进度**，不是用来「绕过」依赖或跳过 Task。是「留痕」，不是「禁止」。

## 3. 产物落盘结构（`docs/`）

```
project-root/
├── CLAUDE.md                         # 工作流主纲（项目根，自动加载）
├── docs/
│   ├── FEATURES.md                   # 【项目级】功能注册表 + 「当前活动功能」指针 ⭐
│   └── features/
│       └── <功能名>/                  # 一个功能 = 一套完整流程
│           ├── SPEC.md               # 【阶段1】需求 + 决策日志 + 边界 + 权衡（单一事实源）
│           ├── ROADMAP.md            # 【阶段2】路线图 + 每 Milestone 详细设计 + Task 表（内联）
│           └── PROGRESS.md           # 【阶段2起】进度驱动器：当前指针 + Task 状态表 + 踩坑 ⭐
└── src/                              # 代码
```

⭐ `FEATURES.md`（项目级）+ 每功能的 `PROGRESS.md` 用**固定文件名、不带日期**——被 agent 按固定路径反复读写。版本/日期放在文件**头部表格**里。模板见 `templates/`。

## 4. 版本控制约定

- **分支**：每个功能一个分支 `feat/<功能名>-M01-<slug>`；Task 在该分支上小步提交。
- **commit 规范**（中文，遵循全局规则）：
  - `feat: 新增 xxx` / `fix: 修复 xxx` / `test: 补充 xxx 测试`
  - `refactor: 重构 xxx` / `docs: 更新 xxx 文档`
- **小步提交**：每个 Task 内按逻辑分多次 commit，不要一个大 commit。
- **PR**：Milestone / 功能完成后开 PR，PR 描述引用 `ROADMAP.md` 验收标准逐条对照。
- ⚠️ **未经用户授权，不要 push 或开 PR。**

## 5. 如何启动

### 全新功能（最常用）
1. 用户说出需求 / 说「访谈」 → 触发**阶段 1 访谈**。
2. agent 深度多轮访谈 → 落盘 `docs/features/<功能名>/SPEC.md` + 注册 `FEATURES.md` → **停下等确认**。
3. 用户确认后说「规划」 → **阶段 2** 产出 `ROADMAP.md` + `PROGRESS.md` → **停下等确认**。
4. 用户粘贴【推进提示词】（或说「推进」） → **阶段 3** 做第一个 Task → 自测 → 更新 PROGRESS → commit → **停下等人工验收**。
5. 用户验收 OK → 再粘同一句【推进提示词】 → 下一个 Task …… 循环到功能完成。

### 现有项目（增量开发）
1. **先探索现有代码**：读 README、CLAUDE.md、目录结构、关键模块。
2. 把对现状的理解写进该功能 `SPEC.md` 的「现状」章节，再谈新需求。
3. 其余同上。

### 切换功能
- 改 `FEATURES.md` 的「当前活动功能」指针，或说「切到 <功能名>」。

### 给 agent 的入口指令
- 新功能："我要做一个功能：<描述>。"（触发访谈）
- 规划："规划 / 拆分 / 出路线图。"
- 推进："推进 / 继续 / next。"（每次同一句，做完一个 Task 即停）

## 6. 阶段产物速查

| 阶段 | 输入（读） | 输出（写） | prompt |
|---|---|---|---|
| 1 理解 | 对话 / 现有代码 | `docs/features/<功能名>/SPEC.md` + `FEATURES.md` 注册 | `phases/1_understand.md` |
| 2 规划 | `SPEC.md` | `ROADMAP.md` + `PROGRESS.md`（初始化，指针→Task 01） | `phases/2_plan.md` |
| 3 执行 | `FEATURES.md`→`PROGRESS.md`（定位当前 Task）+ ROADMAP + SPEC | 代码+测试 + `PROGRESS.md`（✅+commit+移指针） | `phases/3_build.md` |
| 4 验证 | 代码 + 验收标准 | 验证记录（写回 `PROGRESS.md`）+ P0 则回退 🔄 | `phases/4_verify.md` |
| 5 集成 | 已验证的 Task | PR / 发布 + `PROGRESS.md` + `FEATURES.md` 更新 | `phases/5_ship.md` |

---

## 7. 全局禁区

- ❌ 不要假设需求 —— 不清楚就问（见阶段 1）。
- ❌ 不要在 build 阶段重新设计架构 —— 架构问题记到 SPEC 决策日志，回到规划。
- ❌ 不要写「后续考虑 / 可以扩展 / 视情况而定」 —— SPEC 要明确，模糊就是阶段 1 没做完。
- ❌ 不要跳过测试 —— 没有通过的测试，build 就没完成。
- ❌ 不要跳过未 ✅ 的前置 Task，也不要改 `PROGRESS.md` 来「绕过」依赖。
- ❌ 不要自动串接阶段 / 自动做下一个 Task —— 每个闸门都要停下等人。
- ❌ 不要未经确认 push / 开 PR / 删除文件。
